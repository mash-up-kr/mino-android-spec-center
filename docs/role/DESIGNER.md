# 🎨 디자이너 사용 가이드

디자이너는 **spec 컨펌 게이트**를 담당합니다. 검토 중인 스펙을 화면 단위로 확인하고 **승인 / 반려**로 파이프라인을 통과시킵니다.
plan·PR에는 관여하지 않습니다(plan 검증은 PR 리뷰가 흡수). 개발자용 가이드는 [DEVELOPER.md](DEVELOPER.md)를 참고하세요.

> 파이프라인 상태 정의·전이 규칙은 [state-machine.md](../design/state-machine.md), 아래 액션 권한은 [firestore.rules](../../firestore.rules)로도 강제됩니다.

---

## 0. 로그인 & 화면 구성

1. 대시보드 접속 → **GitHub 로그인**(팝업).
2. 최초 로그인 시 **역할 선택** → `디자이너`.
3. 화면 구성
   - **상단 KPI** — 그중 **검토중**이 지금 컨펌을 기다리는 스펙 수
   - **좌측 파이프라인 필터** — 상태별 개수. **`검토중`** 필터로 볼 대상만 추립니다.
   - **중앙 목록** — Feature 검색, 상태/버전 컬럼
   - **우측 상세 패널** — 스텝퍼 · 출처(Figma) · 문서 · **검토 액션**

---

## 1. 검토 대기 확인

좌측 필터 **`검토중`** 또는 상단 KPI **검토중**을 눌러 `spec_in_review` 상태의 스펙만 추립니다.
목록에서 대상 Feature를 선택하면 우측에 상세가 열립니다.

## 2. 스펙 검토

상세 패널의 **`📝 스펙 검토`** → 스펙 미리보기(리뷰 모드)가 열립니다.

- **출처(Figma) 링크**로 원본 화면과 대조합니다.
- 화면 상태·섹션마다 **제목 옆 💬 아이콘으로 인라인 코멘트**를 답니다.
  - 특정 화면/규칙에 정확히 앵커링되어 개발자가 어디를 고쳐야 하는지 명확합니다.
  - 코멘트는 **개별 삭제** 가능하고, 한글 입력(IME) 중 Enter로 잘못 등록되지 않도록 가드되어 있습니다.

## 3. 승인 / 반려

미리보기에서 결정합니다.

- **승인** → `spec_approved`. 개발자의 plan 잠금이 풀립니다.
- **반려** → `spec_changes_requested`. **코멘트가 1개 이상 있어야** 반려됩니다(무엇을 고칠지 없이 반려 불가).

## 4. 보충 코멘트

이미 반려된(`spec_changes_requested`) 스펙에 대해 상태 변경 없이 코멘트만 더 남기고 싶으면 상세의 **`💬 코멘트 추가`**를 사용합니다(decision = comment, 파이프라인 상태는 그대로).

---

## 상태별 디자이너 액션 (한눈에)

| 상태 | 버튼 | 결과 |
|---|---|---|
| `spec_in_review` | **`📝 스펙 검토`** | 미리보기에서 **승인**(→`spec_approved`) / **반려+코멘트**(→`spec_changes_requested`) |
| `spec_changes_requested` | `💬 코멘트 추가` | 상태 변화 없이 보충 코멘트만 기록 |
| 그 외 상태 | (액션 없음) | 개발자/PR 단계 — 디자이너 관여 없음 |

```
        spec_in_review
          │  📝 스펙 검토 (Figma 대조 + 💬 인라인 코멘트)
    ┌─────┴─────┐
  승인          반려 + 코멘트(1개 이상)
    ▼                ▼
spec_approved   spec_changes_requested ──▶ (개발자 수정 후 재요청) ──▶ spec_in_review
 (plan 잠금 해제)        │ 💬 코멘트 추가(상태 유지)
```

> 검토 중(`spec_in_review`)에는 개발자가 spec을 수정할 수 없습니다. 남긴 코멘트는 반려 후 다음 라운드에서 반영됩니다.
> 개발자가 승인 이후 spec을 수정하면 자동 무효화되어 다시 `spec_in_review`로 돌아올 수 있습니다(재검토 시 "지난 검토 이후 변경분" diff 확인 가능).
